Three ways to structure FacilityIQ

All three use AppKit (React + shadcn/ui) + Lakebase Synced Tables. The difference is deployment structure and scope.

A

AppKit + DABs — Full IaC Recommended

One databricks.yml bundle manages the AppKit app, the Lakebase instance, and the pipeline jobs as code. Deploy everything with databricks bundle deploy. Shows off the full Databricks stack to judges.

Pros

  • Full infrastructure-as-code
  • Clean redeploys, consistent state
  • Best demo story for judging
  • DABs + AppKit + Lakebase = full stack

Cons

  • More upfront config (~1 hr)
  • DABs adds learning curve if new to it
B

AppKit app + Notebooks deployed separately

Scaffold the AppKit app with databricks apps init and deploy it manually. Run the LLM pipeline notebooks as a standalone Databricks Workflow from the UI. No DABs needed.

Pros

  • Faster to get first version running
  • Less config overhead
  • Good if team is new to DABs

Cons

  • Manual coordination between pipeline and app
  • Harder to redeploy cleanly
  • Less impressive IaC story for judges
C

AppKit with seed data first, wire real data after

Scaffold AppKit with realistic mock/seed data. Get the full UI polished (search, scorecard, workbench, contradiction alerts). Wire in real Delta/Lakebase data once the pipeline runs. Fastest path to a working demo UI.

Pros

  • Fastest visible progress
  • UI can be perfected independently of data pipeline
  • Easy to iterate on design

Cons

  • Requires two phases — risks not finishing live data wiring
  • Demo may show seed data if pipeline doesn't finish
